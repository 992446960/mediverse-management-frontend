# 开发计划一致性审查

这个 Skill 用来判断“开发计划是否做对需求”，而不只是判断任务是否写得足够多。

核心目标：
- 找出开发计划和产品规划、PRD、API 设计之间的不一致
- 在开发前暴露遗漏项、超范围项、错误假设和高返工风险
- 输出可以直接修订计划的结论，而不是泛泛而谈

## 适用范围

适用于：
- 审查 `doc/前端开发文档/plans/*.md` 下的计划文档
- 审查用户临时写的模块开发计划、TODO 拆解、实施方案
- 用户要求“看看这个计划是否符合产品规划 / PRD / API 设计”
- 用户要求“检查这个模块有没有漏需求 / 漏接口 / 漏验收”

不适用于：
- 纯代码实现质量 review
- 纯 UI 视觉验收
- 从零生成一份完整产品规划

## 默认基线文档

优先使用本项目内以下文档作为审查依据：

- 产品规划：`doc/初始文档/产品规划.md`
- PRD：`doc/初始文档/PRD.md`
- API 设计：`doc/初始文档/API设计.md`
- 计划索引：`doc/前端开发文档/plans/README.md`

按需补充：

- 技术设计：`doc/初始文档/技术设计.md`
- 前端架构与阶段规划：`doc/前端开发文档/前端开发架构与阶段规划.md`
- 前端实施指南：`doc/前端开发文档/前端开发实施指南.md`

如果用户明确指定了别的规划文档、PRD 或接口文档，优先采用用户指定版本，并说明替换了默认基线。

## 审查前动作

### 1. 先定位审查对象

- 如果用户给了计划文件路径，直接读取该文件
- 如果用户只给模块名或 Plan 编号，优先在 `doc/前端开发文档/plans/` 中定位候选文件
- 如果存在多个候选，先向用户确认要审查哪个计划
- 如果用户给的是聊天中的文本计划，先提炼结构，再进入对照审查

### 2. 先提炼计划骨架

在开始给结论前，先从计划中整理出：

- 模块名称与目标范围
- 对应角色与访问权限
- 涉及页面、组件、状态、接口、数据结构
- 前置依赖和并行关系
- 验收项
- 计划明确要做什么
- 计划明确不做什么

如果这些信息在计划里缺失，要把“计划本身信息不完整”列为问题，而不是自行补写。

## 计划文档优先阅读顺序

如果目标是仓库内的 `plan-xx` 文档，优先读这些部分：

1. `## 一、设计文档`
2. `### 关键设计要点`
3. `## 三、计划内容`
4. `## 四、验收效果`

必要时再回到产品规划、PRD、API 设计补证据，不要先无差别通读全部文档。

## 核查维度

### 一、与产品规划对齐

重点检查：

- 计划覆盖的模块是否属于产品规划中的目标范围
- 角色和权限是否符合产品规划中的权限矩阵
- 实体关系是否符合“机构 -> 科室 -> 用户 -> 分身 / 知识库”的层级
- 是否混淆机构级、科室级、个人级能力边界
- 是否遗漏产品规划里明确要求的核心能力
- 是否新增产品规划没有承诺的一期核心能力
- 计划依赖顺序是否符合 `doc/前端开发文档/plans/README.md` 的阶段关系

常见问题信号：

- 普通用户被计划成可管理机构、科室或 API Token
- 科室能力被误写成机构通用能力
- 把“工作台”“我的分身”“数字医生体验”三类场景混写
- 在前置计划未完成时直接依赖未落地的基础设施

### 二、与 PRD 对齐

重点检查：

- 是否覆盖到对应用户故事的主流程
- 任务清单是否足以支撑 PRD 的验收标准
- 是否覆盖默认态、空态、加载态、错误态、权限受限态
- 是否覆盖关键业务规则和异常处理
- 是否遗漏边界条件、非功能要求或安全要求
- 是否存在明显超范围实现

特别注意：

- PRD 中的“角色隔离”“评分触发时机”“上传与处理流程”“搜索追问”“版本对比/回滚”等要求，不能只做表面页面
- 如果计划只有 CRUD 页面，没有覆盖 PRD 里的交互与规则，应判定为缺口
- 如果计划写了 PRD 未定义的大功能，要标记为范围膨胀，而不是默认合理

### 三、与 API 设计对齐

重点检查：

- 计划引用的接口是否在 `doc/初始文档/API设计.md` 中真实存在
- 请求方法、路径、查询参数、请求体、返回结构是否一致
- 是否遵循统一响应包装 `{ code, message, data }`
- 列表接口是否遵循分页结构 `{ total, page, page_size, items }`
- 枚举值、字段名、状态机是否与接口文档一致
- 是否遗漏特殊交互协议，如 SSE、轮询、上传、版本对比、启用/禁用、刷新 Token
- 前端类型定义或页面状态是否建立在不存在的接口字段之上

常见问题信号：

- 计划直接假设接口返回平铺数据，而不是 `data`
- 计划没写分页、筛选、错误码或权限拒绝场景
- 计划使用了 API 文档没有定义的路径或字段
- 计划把轮询接口当成同步接口使用
- 计划忽略 SSE 事件类型或消息拼装规则

### 四、与技术落地约束对齐

只有在计划涉及字段、权限、SSE、数据模型、目录分层时，才补充核对：

- `doc/初始文档/技术设计.md`
- `doc/前端开发文档/前端开发架构与阶段规划.md`
- `doc/前端开发文档/前端开发实施指南.md`

重点看：

- 字段是否和数据模型一致
- 权限隔离是否和技术方案一致
- 目录落点、类型定义、API 封装、状态管理是否符合前端分层
- 是否把后端尚未定义的实现细节写成既定前端方案

## 审查方法

1. 从计划标题、任务清单、验收项中抽取关键词
2. 用关键词去匹配产品规划模块、PRD 用户故事、API 章节
3. 建立对照关系：
   - 计划项 -> 产品依据
   - 计划项 -> PRD 依据
   - 计划项 -> API 依据
4. 标出四类问题：
   - 缺失：文档要求有，但计划没有覆盖
   - 冲突：计划写法与文档直接矛盾
   - 模糊：计划描述不足，无法确认能否满足要求
   - 超范围：计划引入了当前文档基线外的能力
5. 给出严重级别：
   - 严重：会直接导致开发方向错误、接口用错、权限出错或范围跑偏
   - 中等：会导致返工、验收失败、联调受阻或遗漏关键状态
   - 轻微：命名、粒度、引用不精确，但短期可修正

如果发现产品规划、PRD、API 设计三者之间本身存在冲突，要明确指出“源文档冲突”，不要擅自替用户拍板。

## 输出要求

先给结论，再给依据。不要先长篇解释背景。

输出格式：

```md
结论：通过 / 有条件通过 / 不通过

审查对象：
- 计划文件或计划名称

对齐情况：
- 产品规划：对齐 / 部分对齐 / 不对齐
- PRD：对齐 / 部分对齐 / 不对齐
- API 设计：对齐 / 部分对齐 / 不对齐

主要发现：
1. [严重/中等/轻微] 文件或章节
   问题描述
   依据：引用具体文档路径 + 章节
   影响
   建议修订

缺失清单：
- 缺少的任务
- 缺少的验收项
- 缺少的接口或异常场景

通过依据：
- 如果通过，说明为什么判定通过
- 如果是有条件通过，说明必须补齐哪些项

待确认问题：
- 仅列出确实无法从现有文档确定的点
```

## 结论标准

可判定“通过”：

- 计划范围与产品规划一致
- 对应 PRD 用户故事已覆盖主流程和关键验收
- API 使用与字段假设没有明显错误
- 依赖和验收项基本完整

可判定“有条件通过”：

- 主方向正确，但有少量缺失项、边界条件遗漏或验收项不完整
- 修订成本较低，不会改变模块主设计

应判定“不通过”：

- 模块范围与产品规划不一致
- 用户故事主流程未覆盖
- 关键角色权限、接口路径、数据结构假设错误
- 依赖顺序错误，导致计划无法按当前阶段执行

## 工作原则

- 以项目文档为准，不自行发明需求
- 重点审查“计划是否对齐”，不是代替用户写实现代码
- 每个关键结论都要落到具体文档依据
- 如果信息不足，优先指出缺失，不要靠猜测补齐
- 如果用户要求顺手修计划，可以先给审查结论，再提出修订版本
