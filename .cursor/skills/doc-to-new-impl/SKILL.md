# Doc-to-New-Impl Skill：旧规格文档 → 新技术栈规格文档 + 使用示例

## 核心定位

本 Skill 的本质是：

> 以旧规格文档作为**"这个组件要达到什么效果"的需求说明**，
> 以新项目的实际情况作为**"实现环境约束"**，
> 重新跑一遍 code-to-doc 的完整分析过程——
> 只不过这次分析的对象是"如何在新技术栈里实现同等效果"，而不是在读旧代码。

**关键原则：**
- 旧规格文档只告诉你"要做什么、达到什么效果"，不告诉你"怎么做"
- "怎么做"完全按新技术栈的最佳实践重新设计
- Element Plus 有但 Ant Design Vue 没有的功能：用新技术栈自己的方式实现同等效果，不做妥协
- 输出的文档格式与 code-to-doc 完全一致，详细程度完全一致

---

## 工作流程

### Step 1：读取并拆解旧规格文档

收到旧规格文档后，逐节提取以下内容，区分"效果层"和"实现层"：

```
效果层（What）→ 完整保留作为新文档的目标：
  - 组件职责
  - Props 的语义和作用（名称、类型、默认值不变，除非新项目业务字段不同）
  - Emit 事件的触发时机和携带数据的语义
  - 用户交互效果（点击、输入、选择后发生什么）
  - 边界状态（loading/空/禁用/错误的表现）
  - 数据流（数据从哪来、怎么流转、从哪出去）

实现层（How）→ 丢弃，在新技术栈里重新设计：
  - 具体使用了哪个 UI 库组件
  - 该组件的 props/events/slots 配置
  - UI 库提供的内置方法调用（如 ElMessage、ElLoading）
  - v-model 绑定语法
  - 插槽结构
```

---

### Step 2：收集新项目上下文

**必须在此步骤获得以下信息后才能开始写文档：**

| 信息项 | 获取方式 | 若缺失 |
|--------|----------|--------|
| 目标 UI 库及版本 | 用户提供 | 主动询问 |
| 新项目已有组件代码（风格参考） | 用户提供 | 主动询问，这是最重要的参考 |
| 新项目的请求封装方式（若组件涉及 API 调用） | 用户提供 | 询问或假设标准 axios 封装 |
| 新项目的业务字段/接口结构（若与旧项目不同） | 用户提供 | 询问，若相同则跳过 |
| 新项目的 store 使用约定（若组件涉及全局状态） | 用户提供 | 询问 |

收到新项目代码后，提取其编码风格：
- `<script setup>` 还是 Options API
- TypeScript 还是 JS
- 组件内是否抽 composable
- CSS 方案（scoped / tailwind / CSS Modules）
- 命名约定（驼峰/短横线）

---

### Step 3：重新分析，逐节输出新规格文档

**输出格式与 code-to-doc 完全一致**，但每一节都是针对新技术栈重新分析的结果。

处理每一节的规则如下：

---

#### 元信息
```
- 源技术栈：[旧技术栈]（原文保留）
- 目标技术栈：[新技术栈]（更新）
- 组件职责：[完全保留旧文档描述]
- 组件类型：[完全保留]
- 转换说明：本文档基于旧规格文档，针对 [新技术栈] 重新分析实现方案
```

---

#### 一、Props 接口规格
**默认完全保留**（名称/类型/默认值/语义不变）。

例外情况——若用户提供了新项目业务字段与旧项目不同，则按新项目字段更新，并在每个变动项后标注：
```
⚠️ 字段映射：旧项目 userName → 新项目 name
```

---

#### 二、内部状态规格
重新分析在新技术栈中需要哪些内部状态来支撑同等效果。

对于每个状态，完整输出（与 code-to-doc 格式一致）：
```
- 变量名：xxx（沿用旧命名，或按新项目约定调整）
- 类型及初始值：xxx
- 用途：xxx
- 修改时机：xxx
- 对 UI 的影响：xxx
- 新技术栈实现方式：ref() / reactive() / computed() 等
```

**重点处理**：旧技术栈中依赖 UI 库内部状态管理的功能（如 el-form 的表单验证状态），
在新技术栈中需要重新设计状态结构。

---

#### 三、计算属性规格
计算逻辑（What）完全保留，实现方式（How）按新技术栈重写。

若旧文档中某个计算属性依赖了 UI 库特有的数据结构（如 el-table 的 selection），
需重新设计该计算属性的依赖源。

---

#### 四、方法与事件规格

每个方法完整输出，**执行步骤的逻辑语义保留，实现手段按新技术栈重写**：

```
- 名称：xxx
- 触发时机：xxx（保留）
- 入参：xxx（保留）
- 执行步骤（新技术栈实现）：
  1. [逻辑语义保留，但 UI 库 API 替换]
     旧：调用 ElMessage.success('xxx')
     新：调用 message.success('xxx')（import { message } from 'ant-design-vue'）
  2. [若旧步骤依赖 EP 特有功能，用新技术栈等效方案替代，并说明替代理由]
- 对状态的修改：xxx（保留）
- Emit 事件：xxx（保留）
- 副作用：xxx（更新为新技术栈写法）
```

**无法直接映射时的处理规则：**
- 找到 Ant Design Vue 或 Vue3 生态中实现同等效果的最佳方案
- 在步骤中写明方案并解释为何这样设计
- 不妥协、不降级功能，必须达到同等用户体验

---

#### 五、生命周期与副作用规格
生命周期钩子与 UI 库无关，**逻辑完全保留**。

若副作用中涉及 UI 库 API（如 `this.$nextTick` 的特殊用法、UI 库的实例方法），
更新为新技术栈写法。

---

#### 六、UI 结构规格 ⬅ 本节改动最大

**完全重新设计**，不是做名称替换，而是：

1. 理解旧结构每个节点要实现的**视觉效果和交互效果**
2. 找到新 UI 库中实现该效果的最佳组件和配置方式
3. 按 code-to-doc 的层级格式输出新结构

对每个新 UI 库组件，完整说明：
```
<a-xxx> 实现效果：[对应旧文档中该节点的效果描述]
  ├── 使用原因：为什么选这个新组件实现该效果
  ├── 关键 props 配置：
  │     - propA: xxx → 作用：xxx
  │     - propB: xxx → 作用：xxx
  ├── 事件绑定：
  │     - @change="xxx" → 对应旧文档的 @xxx 事件，参数结构变化说明：xxx
  └── 插槽：
        - #xxx → 用于 xxx
```

**Element Plus 有但 Ant Design Vue 无直接对应的功能：**
```
[功能名称]
旧实现：el-xxx 的 xxx 特性
新方案：[完整描述用新技术栈如何实现同等效果]
选择理由：[为什么这样设计]
实现要点：
  1. xxx
  2. xxx
```

---

#### 七、样式规格
- class 命名保留（与业务逻辑绑定的 class）
- UI 库内置 class 替换为新 UI 库等效方式（prop 控制或新 class）
- 若需要额外 CSS 来弥补新 UI 库的样式差异，在此节说明

---

#### 八、数据流图
**完全保留**旧文档描述（数据流与 UI 库无关）。

若业务字段有变化，更新字段名。

---

#### 九、新技术栈实现要点总结

本节为新增节，汇总本次转换中所有关键决策：

```markdown
## 九、新技术栈实现要点总结

### 9.1 组件映射决策
| 旧组件/特性 | 新实现方案 | 决策理由 |
|------------|-----------|---------|
| el-xxx | a-xxx | xxx |
| el-yyy（AntDV 无对应）| 自定义方案：xxx | xxx |

### 9.2 需要开发者关注的差异点
列出所有实现上与旧版本有本质差异的地方，让开发者审查时重点关注：
1. [差异点]：旧版本 xxx，新版本 xxx，用户感知上 xxx
2. ...

### 9.3 验收清单
（从旧文档第九节完整迁移，按新技术栈更新验证方式）
- [ ] [功能点] 验证方式：xxx
```

---

### Step 4：输出使用示例文件（XxxDemo.vue）

独立输出一个 Demo 文件，**穷举所有 props 组合**，每种组合写一个独立示例片段。

**文件结构：**

```vue
<!-- XxxDemo.vue -->
<!-- 组件使用示例文件 - 覆盖所有 Props 组合场景 -->

<script setup lang="ts">
// 只写每个示例涉及的关键响应式变量和事件处理逻辑
// 不写与示例无关的代码
</script>

<template>
  <div class="demo-container">

    <!-- ==================== -->
    <!-- 示例1：基础用法（仅必填 props） -->
    <!-- ==================== -->
    <!-- 说明：[描述这个示例展示什么] -->
    <XxxComponent
      :propA="value1"
      @eventX="handleX"
    />

    <!-- ==================== -->
    <!-- 示例2：开启 [可选功能A]（propB=true） -->
    <!-- ==================== -->
    <!-- 说明：当 propB 为 true 时，组件表现为 xxx -->
    <XxxComponent
      :propA="value1"
      :propB="true"
      @eventX="handleX"
    />

    <!-- 以此类推，穷举所有有意义的 props 组合 -->
    <!-- 边界状态示例：loading、空数据、禁用 -->
    <!-- 组合示例：多个可选 props 同时生效 -->

  </div>
</template>
```

**穷举原则：**
- 每个可选 prop 单独出现一次示例
- 多个相关 prop 组合出现一次示例
- 边界状态（loading=true / 空数据 / disabled）各一个示例
- 每个 Emit 事件至少在一个示例中演示监听和处理

---

### Step 5：输出文件

**文件1**：`[组件名]-新技术栈-spec.md`
**文件2**：`[组件名]Demo.vue`

两个文件都保存至 `/mnt/user-data/outputs/`，用 `present_files` 提供下载。

## 质量自检清单

输出前逐项确认：

**规格文档：**
- [ ] 每节的详细程度与 code-to-doc 输出保持一致
- [ ] Props 的名称/类型/语义与旧文档完全一致（或有明确的字段映射说明）
- [ ] Emit 事件的名称/触发时机/数据语义与旧文档完全一致
- [ ] 第六节每个旧 UI 组件都有明确的新实现方案，无"待定"或"参考旧版"
- [ ] 无法直接映射的功能都给出了完整的替代方案，而非降级处理
- [ ] 第九节汇总了所有关键转换决策

**示例文件：**
- [ ] 每个 Props 都至少在一个示例中出现
- [ ] 每个 Emit 事件都有对应的处理示例
- [ ] 包含 loading、空数据、禁用等边界状态示例
- [ ] 代码片段可直接复制使用，无占位符

## 与 code-to-doc 的协作关系

```
旧项目组件代码
      ↓  code-to-doc（Skill 1）
旧技术栈组件规格文档
（"这个组件要做什么、旧项目怎么做的"）
      ↓
      ↓  doc-to-new-impl（本 Skill）
      ↓  + 新项目上下文（代码风格/技术栈/业务字段）
      ↓
新技术栈组件规格文档        +        XxxDemo.vue
（"新项目怎么实现同等效果"）    （"新项目中如何使用这个组件"）
      ↓
开发者审查 → 新项目实现
```
