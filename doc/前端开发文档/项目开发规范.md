### 4.3 数据加载模式

> 佐证：`src/views/admin/Organizations/index.vue` 第 55-57、178-199、271 行

列表页数据加载统一使用以下模式：

```typescript
// 1. 声明状态
const tableData = ref<Organization[]>([])
const loading = ref(false)
const total = ref(0)

// 2. 加载函数：从 PageFilter / PageTable 获取参数
async function loadData() {
  const params = pageFilterRef.value?.filteParams ?? {}
  const page = pageTableRef.value?.currentPage ?? 1
  const pageSize = pageTableRef.value?.pageSize ?? 20

  loading.value = true
  try {
    const result = await getOrganizations({
      page,
      page_size: pageSize,
      name: (params.name as string) || undefined,
      status,
    })
    tableData.value = result.items
    total.value = result.total
  } finally {
    loading.value = false
  }
}

// 3. 挂载时加载
onMounted(loadData)

// 4. 筛选/分页事件
function onFilterFetch() {
  pageTableRef.value?.resetCurPage(1)
  loadData()
}
function onTableFetch() {
  loadData()
}
```

### 4.7 表单头像设置

> 参考实现：`src/views/admin/Avatars/components/steps/StepInfo.vue` 第 20-74、234-374 行

所有涉及头像上传的表单（分身配置、用户/机构资料等）应与分身设置的头像交互与实现保持一致：

- **API**：统一使用 `uploadAvatar(file)`（`@/api/upload`），上传成功后使用返回的 `url` 写入表单字段（如 `avatar_url`）。
- **交互**：
  - 使用隐藏的 `<input type="file" accept="image/jpeg,image/png,image/gif,image/webp">`，通过 ref 在点击上传区域时触发 `click()`。
  - 上传区域：固定尺寸（如 80×80）圆角卡片、虚线边框；无图时显示加号图标与「点击上传」类文案，有图时显示预览图，悬浮层提供「预览」「更换」操作（图标可用 `EyeOutlined`、`UploadOutlined`）。
  - 上传中显示 `a-spin` 并禁用点击；上传失败时清空预览并恢复空状态。
- **文案与 i18n**：尺寸/格式说明（如建议 512×512、支持 JPG/PNG 等）使用 i18n key，禁止硬编码；与分身设置共用或复用同一套 key（如 `avatar.wizard.avatarUpload`、`avatar.wizard.avatarSizeHint`、`avatar.wizard.avatarFormatHint`）。
- **内存与状态**：本地生成的 `blob:` 预览 URL 在清空头像或组件卸载时调用 `URL.revokeObjectURL()`，避免内存泄漏；表单项绑定上传成功后服务端返回的 URL，不长期持有 blob。

### 4.8 文件大小限制

- 单文件建议 **200–400 行**，最大 **800 行**
- 超出时将子组件、composable 拆分到独立文件
- 函数建议 **< 50 行**

### 5.1 Setup Store 风格

> 佐证：`src/stores/auth.ts` 第 15-102 行

所有 Pinia Store 统一使用 Composition API 风格（Setup Store），**禁止 Options Store**：

```typescript
export const useAuthStore = defineStore(
  'auth',
  () => {
    // ref → state
    const user = ref<User | null>(null)
    const token = ref<string | null>(getToken())

    // computed → getters
    const isLoggedIn = computed(() => !!token.value)
    const isSysAdmin = computed(() => currentRoles.value.includes('sysadmin'))

    // function → actions
    async function login(params: LoginParams) { ... }
    async function logout() { ... }

    return { user, token, isLoggedIn, isSysAdmin, login, logout }
  },
  {
    persist: {
      key: AUTH_STORAGE_KEY,
      pick: ['user'],  // 精确指定持久化字段
    },
  }
)
```

### 5.2 Store 缩进格式

`defineStore` 工厂函数内所有语句统一使用 **4 空格缩进**。

### 5.3 Store 使用方式

```typescript
// 在 <script setup> 中使用
const authStore = useAuthStore()

// 解构响应式属性用 storeToRefs
import { storeToRefs } from 'pinia'
const { currentDeptId, user } = storeToRefs(authStore)

// 方法直接调用
await authStore.login(params)
```

### 5.4 持久化约定

- 使用 `pinia-plugin-persistedstate`，在 Store 第三参数配置
- `key` 使用常量（如 `AUTH_STORAGE_KEY = 'mediverse_auth'`）
- `pick` 精确指定需持久化的字段，**禁止持久化 token**（token 通过 `utils/auth.ts` 独立管理）

## 六、API 调用与 Mock

### 6.1 API 函数组织

> 佐证：`src/api/avatars.ts`

每个业务模块对应 `src/api/` 下一个文件。函数命名遵循 `动词 + 资源名` 模式：

```typescript
import { request } from '@/api/index'
import type { PaginatedData } from '@/types/api'
import type { Avatar, AvatarListParams } from '@/types/avatar'

/** 分页查询分身列表 */
export function getAvatars(params: AvatarListParams) {
  return request.get<PaginatedData<Avatar>>('/avatars', { params })
}

/** 创建分身 */
export function createAvatar(data: CreateAvatarParams) {
  return request.post<Avatar>('/avatars', data)
}

/** 更新分身 */
export function updateAvatar(id: string, data: UpdateAvatarParams) {
  return request.put<Avatar>(`/avatars/${id}`, data)
}

/** 删除分身 */
export function deleteAvatar(id: string) {
  return request.delete<null>(`/avatars/${id}`)
}
```

**约定**：
- 使用 `request.get/post/put/delete/patch` 封装（非原始 `axios`）
- 返回类型通过泛型指定：`request.get<PaginatedData<T>>`
- 每个函数添加中文 JSDoc 注释
- 参数类型和返回类型在 `src/types/` 中定义

### 6.2 `request` 封装

> 佐证：`src/api/index.ts` 第 204-224 行

```typescript
export const request = {
  get<T = any>(url: string, config?: AxiosRequestConfig): Promise<T> {
    return api.get(url, config) as Promise<T>
  },
  post<T = any>(url: string, data?: any, config?: AxiosRequestConfig): Promise<T> { ... },
  put<T = any>(url: string, data?: any, config?: AxiosRequestConfig): Promise<T> { ... },
  delete<T = any>(url: string, config?: AxiosRequestConfig): Promise<T> { ... },
  patch<T = any>(url: string, data?: any, config?: AxiosRequestConfig): Promise<T> { ... },
}
```

响应拦截器自动解包 `ApiResponse.data`，调用方直接拿到业务数据。

### 6.3 错误处理

- 响应拦截器自动处理：业务 `code !== 0` 弹 toast + reject，401 自动刷新 token

```typescript
loading.value = true
```

```typescript
    })
  }),
]
```

**约定**：
- 响应格式与真实后端一致：`{ code: 0, message: 'ok', data: ... }`
- 使用 `delay()` 模拟网络延迟
- 写操作使用 mutable 副本，不修改原始数据
- **禁止**在组件中直接 import mock 数据文件，必须通过 API 层调用
- MSW 仅在 `VITE_ENABLE_MOCK=true` 时启用

### 10.2 权限守卫

> 佐证：`src/router/guards.ts`

路由守卫执行流程：
1. 白名单（`/login`、`/403`、`/404`）直接放行
2. 未登录 → 重定向 `/login?redirect=xxx`
3. 无 user 信息 → `fetchUserInfo()`，失败清除登录态
4. `must_change_pwd` → 强制跳转 `/change-password`
5. `meta.requiredRoles` → 检查角色权限，无权限跳 `/403`

### 10.3 三层权限控制
